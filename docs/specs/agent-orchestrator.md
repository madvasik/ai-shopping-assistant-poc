# Spec: Agent / Orchestrator

## Назначение

Единственный оркестратор системы. Принимает сообщение и историю диалога, управляет выбором flow, последовательно вызывает сервисы, собирает итоговый ответ. Подробный граф выполнения — `docs/diagrams/workflow.md`.

Тексты LLM-промптов собираются через централизованный prompt registry. Оркестратор и вызываемые сервисы используют имя промпта как часть observability-контракта.

## Шаги и правила переходов

1. **Инициализация** — ленивая загрузка Retriever и Knowledge Base (при первом запросе)
2. **Follow-up detection** — если предыдущий ответ содержал `?`, собирается `enhanced_query` из истории, поток идёт сразу в Task Flow
3. **Проверка тематики (LLM)** — если запрос не по теме магазина → немедленный возврат с отбивкой
4. **Классификация типа (LLM)** — `consultation` → Consultation Flow; `task` → Task Flow; ошибка → Task Flow
5. **Consultation Flow** — извлечение сущностей → экспертный ответ → BM25-поиск → фильтр релевантности
6. **Task Flow** — уточняющий вопрос (если нужен и `count < 2`) → список товаров → BM25-поиск → фильтр релевантности

## LLM Prompt Contract

- Каждый LLM-вызов использует именованный промпт из registry
- Для вызова формируются `system_prompt` и `user_prompt`
- В observability-лог пишутся `function`, `prompt_name`, тексты промптов, ответ, токены и длительность

## Stop Conditions

| Условие | Действие |
|---------|---------|
| Запрос не по теме | Немедленный возврат с отбивкой |
| Уточняющий вопрос сгенерирован | Вернуть вопрос; ждать следующего сообщения как follow-up |
| Уточняющих вопросов ≥ 2 | Пропустить уточнение, перейти к поиску |
| Список товаров пуст | Вернуть диагностическое сообщение |
| Exception в любом сервисе | `try/except` → вернуть текст ошибки |

## Retry / Fallback

| Ситуация | Поведение |
|---------|---------|
| LLM exception | Нет retry; дефолтное безопасное значение |
| Пустой BM25-результат | Поиск по всему запросу, top_k=60, head(5) |
| Нет товаров после LLM-фильтра | Показать все BM25-кандидаты (head 5) |

## Ограничения

LLM-вызовы синхронные внутри async FastAPI — при высокой нагрузке блокируют event loop. Для production нужен async LLM-клиент.
