# Product Proposal: AI Shopping Assistant

## Обоснование идеи

### Описание прикладной задачи

Автоматизация консультаций и подбора товаров в онлайн-магазине строительных товаров. Система помогает пользователям быстро находить нужные товары, получать экспертские советы и рекомендации на основе их запросов. Для задач пользователей система собирает корзины товаров "под ключ" - полный набор необходимых товаров от основных инструментов до вспомогательных материалов.

### Проблема

* Высокая нагрузка на консультантов при росте объёма запросов
* Субъективность в рекомендациях товаров без понимания контекста запроса пользователя
* Долгий цикл от запроса пользователя до получения персонализированных рекомендаций
* Неоднозначные запросы требуют множественных уточняющих вопросов от консультанта

---

## Цель и метрики

### Продуктовые метрики

* **Среднее время ответа:** < 10 секунд (p95 < 20 секунд)
* **Релевантность рекомендаций товаров:** > 75% релевантности (оценка экспертом)
* **Удовлетворенность пользователей:** > 4.0/5.0 (опрос после взаимодействия)

### Агентские метрики

* **Точность классификации интентов:** > 90%
* **Успешность обработки запросов без ошибок:** > 90%

### Технические метрики

* **Latency p95:** < 20 секунд для полного цикла обработки запроса
* **Throughput:** > 10 запросов/минуту
* **Стоимость запроса к LLM:** зависит от выбранной модели LLM (оценка для PoC: < 1 руб/запрос)

## Потенциальные сценарии использования

### Основные сценарии

1. **Consultation Flow (Флоу консультации)**
   - Пользователь задает вопрос о товарах (например: «Что лучше для покраски стен, кисточка или валик?»)
   - Бот генерирует экспертский ответ на вопрос через LLM на основе знаний о строительных товарах
   - Параллельно выделяет сущности из запроса через LLM (например: «краска, кисточка, валик»)
   - Выполняет поиск товаров в каталоге (лексический BM25 по текстовым полям карточек) по выделенным сущностям
   - На выходе: экспертский ответ на вопрос + ссылки на найденные товары

2. **Problem Solving Flow (Флоу решения задачи)**
   - Пользователь описывает задачу (например: «Хочу повесить шторы»)
   - Бот анализирует задачу через LLM и задает уточняющие вопросы для критически важных параметров (например: «Из какого материала стена?»)
   - После получения ответов определяет полный набор необходимых товаров через LLM
   - Выполняет поиск в каталоге (лексический BM25) по каждому необходимому товару
   - Формирует корзину товаров под ключ (от основных инструментов до вспомогательных материалов, например: от дрели до стремянки, от строительного карандаша до дюбелей)
   - На выходе: полный набор товаров для решения задачи, организованный по категориям

### Edge-кейсы

1. **Неоднозначные запросы пользователей**
   - Система определяет, что нужны уточняющие вопросы
   - Задает до 2 уточняющих вопросов для критически важных параметров
   - Обрабатывает запрос после получения ответов

2. **Отсутствие товаров в каталоге**
   - Система обнаруживает, что товары не найдены
   - Выполняет graceful degradation: объясняет причину, дает общие рекомендации
   - Не показывает ошибку пользователю

3. **Ошибки LLM**
   - Система обнаруживает ошибку или неожиданный ответ от LLM
   - Выполняет fallback на базовые ответы или повторный запрос
   - Логирует ошибку для анализа

4. **Проблемы с производительностью**
   - При высокой нагрузке система может испытывать задержки
   - Реализованы базовые механизмы rate limiting (для PoC)
   - Мониторинг производительности через логирование

5. **Запросы вне тематики магазина**
   - Система проверяет релевантность запроса каталогу через LLM
   - Фильтрует запросы, не относящиеся к строительным товарам
   - Объясняет пользователю, что вопрос не относится к тематике магазина

## Ограничения

### Технические ограничения

* **SLO:** доступность 95% (для PoC, не production-ready)
* **Latency p95 (сквозной цикл чата):** целевое значение < 20 секунд; фактическое время сильно зависит от LLM API и числа последовательных вызовов
* **Зависимость от внешнего LLM API:** возможны таймауты или недоступность API
* **Ограничения по размеру каталога:** текущая SQLite-база каталога и in-memory BM25 индекс, ограничения на количество товаров для быстрого поиска в рамках одного инстанса PoC
* **Ограничения по интерфейсам:** Streamlit — вспомогательная панель (логи/история), не целевой канал для массового пользовательского трафика; публичный чат — виджет/API с собственными лимитами и настройкой доменов

### Операционные ограничения

* **Бюджет:** ограничения на количество запросов к LLM (для PoC)
* **Масштабирование:** для роста нагрузки на пользовательский чат потребуется усиление слоя виджета/API и бэкенда; панель Streamlit остаётся инструментом оператора, а не точкой масштабирования клиентского трафика
* **Мониторинг:** базовое логирование запросов к LLM, нет продвинутого мониторинга и алертинга

## Архитектурный набросок

### Модули

* **Модуль классификации интентов** - определение типа запроса пользователя (task/consultation)
* **Модуль проверки релевантности** - проверка релевантности вопроса тематике сайта
* **Модуль поиска товаров (retrieval)** — лексический поиск (BM25) по текстовым полям каталога
* **Модуль генерации ответов** - генерация экспертских ответов через LLM
* **Модуль анализа задач** - анализ задач пользователя и определение необходимых товаров
* **Модуль логирования** - логирование запросов к LLM и мониторинг
* **Слой чата (виджет + HTTP API)** — приём сообщений пользователя, сессионный контекст диалога, отдача ответов в UI
* **Панель наблюдения (Streamlit)** — просмотр истории вызовов LLM и логов для PoC

### Интеграции

* **LLM API** — классификация интентов, проверка релевантности, генерация ответов, анализ задач, извлечение сущностей
* **SQLite каталог товаров** — источник карточек товаров для BM25-поиска (таблица `products`)
* **HTTP API чата** — связь виджета/демо-страницы с оркестрацией и поиском
* **Streamlit** — панель оператора (логи, история запросов к модели), не замена клиентскому чату

## Потенциальный data flow

### Consultation Flow (Флоу консультации)

```
Пользователь → Чат (виджет / демо-страница) → HTTP API
         ↓
Intent Classifier (LLM) → [consultation]
         ↓
┌─────────────────────────────────────┐
│ Knowledge Base (LLM)                │ → Генерация ответа на вопрос
│ Entity Extraction (LLM)             │ → Выделение сущностей (товаров)
└─────────────────────────────────────┘
         ↓
Product Search (BM25) → Поиск товаров по выделенным сущностям
         ↓
Display Results (UI) → Отображение ответа + ссылки на товары
```

### Problem Solving Flow (Флоу решения задачи)

```
Пользователь → Чат (виджет / демо-страница) → HTTP API
         ↓
Intent Classifier (LLM) → [task]
         ↓
Task Analyzer (LLM) → Анализ задачи
         ↓
[Уточняющие вопросы?] → Вопросы для критических параметров
         ↓                    (например: "Из какого материала стена?")
Пользователь отвечает
         ↓
Task Analyzer (LLM) → Определение полного набора товаров
         ↓                    (от основных инструментов до вспомогательных)
Product Search (BM25) → Поиск по каждому товару отдельно
         ↓
Relevance Check (LLM) → Проверка релевантности товаров
         ↓
Display Results (UI) → Корзина товаров под ключ, организованная по категориям
```

### Делегирование LLM/Agent

**Что делегируется LLM/Agent:**
* Классификация интентов пользователя
* Извлечение названий товаров из запроса
* Генерация экспертских ответов
* Анализ задач и определение необходимых товаров
* Проверка релевантности запросов и товаров

**Что НЕ делегируется LLM:**
* Поиск товаров в каталоге — детерминированный лексический поиск (BM25) по индексируемым полям карточек, без «галлюцинаций» списка SKU
* Отрисовка чата и демо-страницы, раздача статики виджета
* Хранение каталога в SQLite и долговременная персистентная память диалогов вне рамок одной серверной сессии PoC
