# System Design: AI Shopping Assistant PoC

## 1. Ключевые архитектурные решения

| Решение | Выбор | Обоснование |
|---------|-------|-------------|
| LLM-провайдер | OpenAI API (`gpt-4o-mini` по умолчанию) | Chat Completions; баланс качества/стоимости; поддержка русского языка |
| Поиск товаров | BM25 in-memory | Детерминированный, без галлюцинаций; достаточен для PoC |
| Каталог | SQLite (read-only) | Zero-ops; легко заменяется в production |
| Session state | In-memory dict | Нет инфраструктурных зависимостей; потеря при перезапуске приемлема для PoC |
| Serving | FastAPI + uvicorn | Async, производительный |
| Observability | SQLite logging + Streamlit | Нулевой порог входа для PoC; в логах — токены и оценка стоимости LLM-вызовов (см. `docs/specs/observability-evals.md`) |
| Intent routing | LLM-классификатор (2 класса) | Гибкость к формулировкам; точность > 90% |
| Relevance gate | LLM-фильтр после BM25 | Отсеивает ложные совпадения поиска |
| Frontend | Embeddable JS-виджет (ES5) | Встраивается в любой сайт без сборщика |

**Что намеренно не делегируется LLM:** поиск по каталогу, сессионная логика, rate limiting, domain ACL.

---

## 2. Модули и их роли

| Модуль | Роль |
|--------|------|
| Widget API (FastAPI) | Сессии, rate limit, domain ACL, роутинг запросов |
| Chat Orchestrator | Главный execution flow: выбор ветки, вызов сервисов, сборка ответа |
| Intent & NLU | Проверка тематики, классификация типа запроса, извлечение сущностей, фильтр релевантности |
| Knowledge Base | Генерация экспертного ответа для consultation-flow |
| Task Analyzer | Уточняющие вопросы и определение списка товаров для task-flow |
| Prompt Registry | Централизованное хранение и сборка system/user промптов для LLM-сервисов |
| BM25 Retriever | In-memory поиск по каталогу (загружается из SQLite один раз) |
| LLM Logger | Callback-based запись всех LLM-вызовов в SQLite |
| Streamlit Dashboard | Просмотр LLM-логов, latency, токенов и оценочной стоимости (USD) по LLM-вызовам и по пользовательским запросам |

## 3. Основной workflow

Подробная схема — в `docs/diagrams/workflow.md`. Кратко:

**Consultation flow** (вопросы, сравнения):
`Запрос → проверка тематики → классификация → извлечение сущностей + генерация ответа → BM25-поиск → фильтр релевантности → ответ`

**Task flow** (задачи типа «поклеить обои»):
`Запрос → проверка тематики → классификация → уточняющий вопрос (если нужен) → список товаров → BM25-поиск → фильтр релевантности → ответ`

---

## 4. State / Memory / Context

- **Сессия** — хранится in-memory: идентификатор виджета, хост страницы, история сообщений
- **История диалога** — максимум 20 сообщений (LIFO), теряется при перезапуске
- **Межсессионная память** — отсутствует намеренно (privacy, PoC scope)
- **Context budget** — в LLM передаётся не более 5 последних сообщений; для уточнений — полная история
- **Follow-up** — при ответе на уточняющий вопрос оркестратор собирает объединённый запрос из исходной задачи + вопроса + ответа

---

## 5. Retrieval-контур

SQLite → DataFrame → BM25-индекс (in-memory, загружается один раз) → поиск top_k=60 → BM25 score + бонус за полное совпадение слов → фильтр score > 0.3, top 3 на категорию → LLM-фильтр релевантности → итоговые карточки товаров

Только лексический поиск; нет стемминга, нет векторного поиска. Подробнее — `docs/specs/retriever.md`.

## 6. Tool / API-интеграции

### LLM-вызовы (OpenAI API)

| Назначение | max_tokens | Fallback при ошибке |
|-----------|:----------:|---------------------|
| Проверка тематики запроса | 10 | Считать «не по теме» |
| Классификация типа запроса | 10 | Считать «задача» |
| Извлечение товаров из запроса | 200 | Пустой список |
| Фильтр релевантности товаров | 10 | Все товары релевантны |
| Генерация экспертного ответа | 1000 | `None` → резервный текст в коде |
| Уточняющий вопрос | 150 | None (пропустить) |
| Список товаров для задачи | 2000 | Пустой список |

Модель задаётся через `OPENAI_MODEL` (default: `gpt-4o-mini`). Подробные контракты — `docs/specs/tools-apis.md`.

System и user промпты собираются через централизованный registry. В логах LLM-вызовов сохраняется `prompt_name` вместе с текстом промпта и ответом модели.

### Внутренний REST API

| Endpoint | Назначение |
|----------|-----------|
| `POST /api/session` | Создание сессии |
| `POST /api/chat` | Отправка сообщения |
| `GET /health` | Health check |

## 7. Failure Modes, Fallback и Guardrails

| Ситуация | Поведение |
|----------|-----------|
| OpenAI API недоступен | Каждая LLM-функция возвращает безопасный дефолт (см. таблицу выше) |
| LLM вернул неожиданный формат | Парсинг с валидацией → дефолт при несоответствии |
| BM25 не нашёл ничего | Fallback: поиск top_k=60 по всему запросу |
| Запрос не по теме магазина | Фиксированный текст-отбивка |
| Rate limit превышен | HTTP 429 + `retry_after` |
| Неизвестная сессия | Авто-создание для localhost; HTTP 400 для остальных |

**Guardrails:** бот не выполняет write-операций с каталогом или заказами; domain whitelist ограничивает встраивание виджета; `max_tokens` ограничивает вывод LLM (защита от prompt injection).

## 8. Технические и операционные ограничения

| Параметр | Значение |
|---------|---------|
| Latency p95 (сквозной) | < 20 сек (3–7 последовательных LLM-вызовов по 1–3 сек каждый) |
| Throughput | > 10 req/min (один инстанс) |
| Rate limit | 30 req / 5 мин на сессию |
| Стоимость запроса | < 1 руб. (оценка для PoC-трафика) |
| SLO доступности | 95% (PoC, не production) |
| Масштабирование | Один инстанс; нет distributed state |
| Зависимость | При недоступности OpenAI большинство функций деградируют |
